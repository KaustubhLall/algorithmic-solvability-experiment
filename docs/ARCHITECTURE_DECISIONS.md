## Pre-Implementation Decisions (from design phase)

These decisions were made during planning and are captured here for completeness.

### ADR-001: Use synthetic tasks with known ground-truth algorithms

- **Date:** 2025-03-25
- **Task:** Design phase
- **Status:** ACCEPTED
- **Context:** Need to evaluate whether models learn the *algorithm*, not just correlations. Requires knowing the true answer for any input.
- **Decision:** All tasks are generated by reference implementations (Python functions). Every label is verified against the reference at generation time.
- **Rationale:** This is the only way to guarantee label correctness and support out-of-distribution testing against a ground truth.
- **Consequences:** All tasks must have a computable reference algorithm. Tasks where the "algorithm" is unknown or approximate cannot be used.

---

### ADR-002: Two parallel task tracks (Sequence and Classification)

- **Date:** 2025-03-25
- **Task:** Design phase
- **Status:** ACCEPTED
- **Context:** Sequence-only tasks miss an entire family of algorithmically solvable problems that look like tabular ML (rule-based classifiers with mixed inputs).
- **Decision:** Maintain two tracks: Sequence (S-tiers, variable-length token sequences) and Classification (C-tiers, fixed-width mixed numerical/categorical tabular inputs).
- **Rationale:** Broadens the scope of the framework without conflating fundamentally different input representations.
- **Consequences:** Model Harness (SR-5) must handle two distinct encoding strategies. DSLs are separate per track (SR-9, SR-10).

---

### ADR-003: Validation-first development order

- **Date:** 2025-03-25
- **Task:** Design phase
- **Status:** ACCEPTED
- **Context:** ML pipelines can silently produce wrong results (label leakage, non-determinism, off-by-one in splits) that only become visible much later.
- **Decision:** Each shared resource (SR) has a corresponding validation procedure (V) that must pass immediately after the resource is built, before proceeding to the next task.
- **Rationale:** Catching bugs early is far cheaper than discovering them after running 20 experiments.
- **Consequences:** Slightly slower initial development, but much higher confidence in experiment results.

---

### ADR-004: Labels are never noisy — noise is input-only

- **Date:** 2025-03-25
- **Task:** Design phase
- **Status:** ACCEPTED
- **Context:** Need to test model robustness to input perturbation without corrupting the ground truth.
- **Decision:** Noise is added to input features only, after label assignment. Labels always reflect the clean input. Noisy inputs therefore have "stale" labels — intentional.
- **Rationale:** Preserves label integrity. Ensures the reference algorithm remains verifiable. Makes noise robustness a well-posed test.
- **Consequences:** Noise injection happens as a post-processing step in the Data Generator (SR-3), not inside the reference algorithm.

---

### ADR-005: Solvability verdict uses a five-label scale

- **Date:** 2025-03-25
- **Task:** Design phase
- **Status:** ACCEPTED
- **Context:** A binary "solvable/not solvable" verdict loses too much nuance.
- **Decision:** Use STRONG / MODERATE / WEAK / NEGATIVE / INCONCLUSIVE labels based on criteria in `EXPERIMENT_DESIGN.md` Section 9.4.
- **Rationale:** Captures partial success, calibrates expectations, and surfaces inconclusive cases that need more investigation.
- **Consequences:** Report Generator (SR-8) must implement verdict logic exactly per Section 9.4. EXP-D5 validates calibration.
